**Use Case Example:**
- **View**: All accounts (user can see everything)
- **Create**: Only specific accounts (user can create payments from accounts 1 & 2)
- **Approve**: Only one account (user can only approve payments from account 1)

**Behavior:**
- Only shows accounts relevant to the service (e.g., DDA accounts for ACH)
- Accounts are from profile's enrolled accounts
- Shows account name/description for easy identification
- Search filters the visible accounts
- "Per Action" is the most granular option for fine-grained control

## State Management

### Initial Load (Edit Mode)

When loading a user's permissions:

1. Fetch user's assigned roles from User aggregate
2. Fetch explicit PermissionPolicy records for user subject
3. Map to UI state:
   - If SERVICE_ADMIN role â†’ select "Service Administrator"
   - Else â†’ select "Specific Permissions" and check relevant roles
   - If custom policies exist â†’ expand Advanced Permissions and populate

### Save Flow

1. Validate selections
2. If Service Administrator or Security Admin â†’ show confirmation modal
3. On confirm:
   - Delete existing user-specific PermissionPolicy records
   - Update user roles in User aggregate
   - Create new PermissionPolicy records for custom permissions
4. Show success toast

## Summary

### Key Behaviors

1. **Permission Level Radio**: Service Administrator vs Specific Permissions
2. **Standard Roles**: Multi-select checkboxes for common role patterns
3. **Advanced Permissions Toggle**: Reveals service-level configuration
4. **Service Group Accordion**: Expand to see individual services
5. **Service Permission Mode**: All vs Individual per service
6. **Account Scope**: Three options when applicable:
   - **All Accounts**: Access to all accounts for all actions
   - **Selected**: Specific accounts apply to all actions for the service
   - **Per Action**: Each action (View/Create/Approve) can have different account access
7. **Confirmation Modal**: Required for admin-level permissions

### Policy Generation

The UI generates policies following the permission-policy-design.md structure:
- Subject: `user:{userId}`
- Action: Hierarchical pattern (e.g., `payments.ach-payments.*.create`)
- Resource: `*` or comma-separated account URNs
- Effect: Always `ALLOW` (DENY managed via API)
